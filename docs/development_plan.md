# 开发设计与计划文档

本文档详细记录微信聊天记录读取器的设计思路、架构细节、开发阶段计划及风险应对策略。

## 1. 系统设计

### 1.1 总体架构
系统采用 **"控制-感知-处理"** 循环模式：
1.  **控制 (Controller)**: 驱动微信窗口滚动。
2.  **感知 (Vision)**: 获取当前可视区域截图。
3.  **处理 (Processor)**: 图像拼接、OCR 识别、结构化解析。

### 1.2 核心模块设计

#### A. 窗口控制模块 (WindowController)
-   **目标**: 稳定定位微信窗口，获取聊天列表区域坐标。
-   **技术**: `uiautomation` + `pywin32`。
-   **功能**:
    -   `find_window()`: 查找微信主窗口句柄。
    -   `get_chat_area()`: 获取右侧聊天详情页的坐标区域 (Rect)。
    -   `scroll_up()`: 模拟鼠标滚轮向上滚动指定距离。

#### B. 视觉捕获模块 (ScreenCapturer)
-   **目标**: 高效获取指定区域的图像数据。
-   **技术**: `mss` (比 PyAutoGUI 快) 或 `PIL.ImageGrab`。
-   **策略**:
    -   支持 DPI 感知，确保截图与物理像素 1:1 对应。
    -   缓存最近 N 帧截图用于拼接比对。

#### C. 内容解析模块 (ContentParser)
-   **目标**: 从图像中提取有意义的聊天数据。
-   **流程**:
    1.  **预处理**: 灰度化 -> 二值化 -> 降噪。
    2.  **区域分割**: 识别头像位置，区分“我发送的”和“对方发送的”消息（基于气泡颜色或位置）。
    3.  **OCR**: 对文本区域调用 Tesseract。
    4.  **非文本处理**: 识别图片缩略图、文件图标等。

#### D. 滚动去重模块 (ScrollDeduplicator)
-   **难点**: 滚动后的新视图与旧视图存在重叠。
-   **算法**:
    -   使用 OpenCV 的模板匹配 (Template Matching) 或特征点匹配 (ORB/SIFT) 寻找上一帧底部在当前帧的位置。
    -   计算 `offset` (像素偏移量)。
    -   只对新出现的区域进行 OCR，减少计算量并避免重复。

### 1.3 数据模型
存储格式建议使用 SQLite，表结构示例：
```sql
CREATE TABLE messages (
    id INTEGER PRIMARY KEY,
    session_id TEXT,       -- 会话标识
    sender TEXT,           -- 发送者 (Me/Others/Name)
    type TEXT,             -- text/image/file/system
    content TEXT,          -- 文本内容或文件路径
    timestamp DATETIME,    -- 识别出的时间
    raw_image_path TEXT    -- 原始截图路径(可选)
);
```

## 3. 技术难点与风险应对

| 难点 | 风险描述 | 应对策略 |
| :--- | :--- | :--- |
| **DPI 适配** | 截图坐标偏移，导致截取区域错误 | 使用 `ctypes` 设置 Process DPI Aware; 使用相对坐标计算。 |
| **滚动重叠** | 无法精确控制滚动像素，导致消息重复或遗漏 | 放弃固定像素滚动，采用"滚动+图像匹配"闭环控制，确认重叠区域。 |
| **背景干扰** | 聊天背景图或气泡颜色影响 OCR | 强制图像二值化；训练 Tesseract 专门识别微信字体（可选）。 |
| **微信风控** | 频繁操作可能触发限制 | 增加随机等待间隔 (Random Sleep); 限制每日采集量。 |
